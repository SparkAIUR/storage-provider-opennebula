# v0.6.0 RC: AttachmentIntent design notes

This note captures the next major attach/detach reliability direction for the SparkAI OpenNebula CSI driver.

The immediate priority is still to fully recover and reconcile the current production cluster state with the `v0.5.11` hotfix line. Once that incident is closed, the `v0.6.0` work should focus on replacing reactive attach/detach behavior with an explicit attachment-intent reconciler.

## Current context

The `v0.5.x` cycle added several targeted controls around OpenNebula hotplug behavior:

- per-node hotplug queueing
- node-health-aware hotplug pause
- local RWO same-node restart reuse
- ConfigMap-driven maintenance mode
- stale `VolumeAttachment` reconciliation
- "already absent" detach handling
- local device missing reports
- same-node detach/reattach recovery when OpenNebula reports a disk attached but the guest does not discover it
- read-only host-artifact quarantine for local `fs_lvm_ssh` attach failures such as stale `lv-one-<vm>-<disk>` LVs

These changes make the current driver safer, but the core model is still mostly RPC-driven:

- `ControllerPublishVolume` receives an attach request and usually mutates OpenNebula immediately.
- `ControllerUnpublishVolume` receives a detach request and usually mutates OpenNebula immediately.
- Recovery logic then reacts when Kubernetes, OpenNebula, and node-side device state diverge.

That model works during normal pod churn, but it is fragile during Talos/Kubernetes upgrades because node reboots, kubelet restarts, OpenNebula HOTPLUG state, and Kubernetes rescheduling can overlap.

## Problem statement

For local RWO volumes, the lowest-risk outcome during maintenance is usually to keep the volume on the same node and wait for that node to return. The current CSI RPC model can instead see a detach request followed by a cross-node attach request and treat those as independent commands.

That can cause:

- unnecessary detach/attach cycles while a VM is booting
- guest SCSI LUN churn
- device discovery timeouts even when OpenNebula believes the disk is attached
- retry loops between Kubernetes attach/detach-controller and the CSI controller
- local RWO PVCs drifting away from their last known-good node during rolling upgrades

The driver needs a central decision point that understands desired state, observed state, operation history, and maintenance context before choosing a physical OpenNebula mutation.

## Proposed strategy

Build an `AttachmentIntent` reconciler.

Instead of treating each CSI attach/detach RPC as a standalone imperative action, the controller should maintain a durable intent ledger and reconcile that intent against observed state.

CSI RPCs still return CSI-compliant responses, but their main job becomes updating intent and scheduling reconciliation. The reconciler chooses the lowest-risk action that satisfies the current desired state.

## Design principles

- Prefer no-op over mutation when the current attachment is already useful.
- Prefer same-node reuse over detach/reattach.
- Prefer waiting over cross-node movement for local RWO during node maintenance.
- Treat OpenNebula attachment and guest device discovery as separate phases.
- Make every physical mutation traceable to a recorded intent generation.
- Make stale operations easy to identify and discard.
- Keep CephFS/RWX paths out of local RWO recovery logic.
- Preserve CSI semantics: retryable conditions should return retryable errors such as `Unavailable` or `Aborted`.

## AttachmentIntent model

Initial implementation can use an internal ConfigMap-backed ledger. Once the behavior is stable, promote it to a CRD.

Candidate object shape:

```yaml

kind: OpenNebulaAttachmentIntent
metadata:
  name: <pv-name-or-volume-hash>

  volumeID: pvc-...
  pvName: pv-...
```

## State inputs

The reconciler should merge state from these sources:

- `VolumeAttachment` objects
- PV/PVC annotations and access modes
- active pods referencing the PVC
- node readiness and unschedulable state
- node drain or upgrade taints
- `opennebula-csi-hotplug-state`
- `opennebula-csi-node-device-state`
- sticky attachment state
- OpenNebula VM disk attachments
- OpenNebula VM LCM/HOTPLUG state
- host-local storage artifacts such as `lv-one-<vm>-<disk>` on LVM/SSH system datastores
- node-side stage/session health

### Incident insight: five-source attachment truth

The May 2026 attach/detach incident exposed a gap that v0.6 must model directly: "attached" is not a single truth. The reconciler needs to compare five independent layers before deciding to mutate anything:

- Kubernetes desired state: pods, PVCs, PVs, `VolumeAttachment` objects, and scheduler intent
- OpenNebula metadata: image `STATE`, `RUNNING_VMS`, `VMS`, and VM template `DISK` records
- hypervisor reality: libvirt/QEMU block devices such as `virsh domblklist`
- guest reality: Talos/Linux SCSI devices, `/dev/disk/by-id`, CSI staging sessions, and mount table state
- host storage reality: local LVM artifacts, especially `lv-one-<vm>-<disk>` on `fs_lvm_ssh` system datastores

The critical failure mode was OpenNebula DB/image metadata claiming ownership while libvirt and the guest no longer had the disk. In that state, repeated CSI detach attempts can fail because the hypervisor target is already absent, and repeated attach attempts can fail because OpenNebula still considers the image owned by the old VM. A second failure mode appeared after DB metadata cleanup: fresh attaches reused the same VM disk slot, but a stale host-side LV still existed and blocked every attach before libvirt saw the disk.

v0.6 should treat these as first-class blocked states:

- `OpenNebulaMetadataDrift`: image/VM metadata owner conflicts with Kubernetes desired node
- `HypervisorAbsentDBAttached`: OpenNebula template/image metadata says attached, but the hypervisor target is absent
- `GuestAbsentOpenNebulaAttached`: OpenNebula says attached and hypervisor may be attached, but the guest cannot discover the device
- `HostArtifactConflict`: OpenNebula is ready to attach to a VM/disk slot, but host-local storage artifacts such as `lv-one-161-2` block the attach before libvirt attachment
- `HostArtifactInspectNeeded`: OpenNebula error output is consistent with a local host attach failure, but the driver needs a host-artifact observation from an external inspector before selecting a repair action
- `ExternalRepairRequired`: the driver has enough contradictory evidence that continuing hotplug mutation is likely to extend the outage

The reconciler should not blindly alternate detach and attach in these states. It should quarantine the affected intent, surface exact repair targets, and wait for explicit evidence that metadata and runtime state agree again.

The v0.5.12 metadata drift guard and v0.5.13 host-artifact quarantine are tactical versions of this rule. The v0.6 intent reconciler should make them durable, observable, and central to every local RWO attach/detach decision.

## State machine

Suggested high-level phases:

- `Unknown`
- `PendingAttach`
- `AttachInProgress`
- `Attached`
- `DevicePending`
- `Staged`
- `MaintenanceHeld`
- `PendingDetach`
- `DetachInProgress`
- `Detached`
- `RecoveryNeeded`
- `RecoveryInProgress`
- `HostArtifactBlocked`
- `Blocked`

Important transitions:

- `PendingAttach -> Attached`
  - OpenNebula reports the disk attached to the desired node.
- `Attached -> DevicePending`
  - OpenNebula attachment exists, but node-side serial/device discovery has not been confirmed.
- `DevicePending -> Staged`
  - node plugin reports device discovered and staging succeeded.
- `DevicePending -> RecoveryNeeded`
  - repeated missing-device reports exceed threshold.
- `RecoveryNeeded -> RecoveryInProgress`
  - reconciler starts same-node detach/reattach.
- `Attached -> MaintenanceHeld`
  - node is upgrading, unschedulable, NotReady, or maintenance mode is active.
- `MaintenanceHeld -> Attached`
  - node returns and same-node desired state still exists.
- `MaintenanceHeld -> PendingDetach`
  - maintenance exits and Kubernetes no longer desires the volume on the held node.

## Action scoring

The reconciler should choose the lowest-risk action that can make progress.

Suggested action cost order:

1. `ObserveOnly`
2. `ClearStaleState`
3. `ReuseSameNodeAttachment`
4. `HoldForMaintenance`
5. `WaitForNodeReady`
6. `SameNodeDeviceRescan`
7. `SameNodeDetachReattach`
8. `HostArtifactInspect`
9. `PhysicalDetach`
10. `CrossNodeAttach`

For local RWO, `CrossNodeAttach` should be unavailable during maintenance and should require explicit evidence that the old node no longer owns the volume or that the hold policy has expired.

## CSI RPC behavior

### ControllerPublishVolume

On publish:

- resolve runtime context
- update desired intent to the requested node
- if the volume is already attached to the requested node, return the current publish context
- if host-local artifacts are known to block the predicted VM/disk slot, return `FailedPrecondition` with the exact VM ID, disk ID, target, and LV name
- if the volume is attached to the last-good node and maintenance/upgrade context is active, return `Unavailable`
- if cross-node movement is allowed, let the reconciler enqueue detach/attach as one ordered operation
- include the intent generation in logs/events

### ControllerUnpublishVolume

On unpublish:

- update intent that Kubernetes no longer wants the volume on the node
- do not physically detach if:
  - the same PVC is still desired by an active pod on the same node
  - maintenance mode is active
  - the node is NotReady but expected to return
  - OpenNebula already reports the disk absent
- otherwise enqueue a detach reconciliation

### NodeStageVolume

On stage:

- report device discovery success by serial/target
- clear missing-device report
- update intent status to `DeviceDiscovered` or `Staged`

On repeated device discovery failure:

- write missing-device report
- update intent status to `DevicePending` or `RecoveryNeeded`
- do not force cross-node scheduling as a side effect

## Maintenance semantics

The current ConfigMap-driven maintenance mode should become an input to the intent reconciler.

When maintenance is active:

- local RWO volumes should get hard same-node stickiness
- detach should become a hold unless the volume is already absent
- cross-node attach should be rejected with `Unavailable`
- readiness should mean "the controller has prepared eligible intents", not "all nodes are healthy"

When maintenance exits:

- release holds gradually
- avoid a thundering herd of detach operations
- preserve last-good node metadata until a new successful stage confirms a different node

## Recovery semantics

The `v0.5.11` local device recovery path should become one reconciler action:

`SameNodeDetachReattach`

Eligibility:

- backend is `local`
- volume is single-writer
- not CephFS/RWX
- desired node equals reporting node
- Kubernetes node is Ready
- OpenNebula VM is Ready
- recovery attempts are under the cap
- cooldown has elapsed

Non-goals:

- do not move the volume to a different node
- do not detach a volume from an unhealthy node just because Kubernetes retried elsewhere
- do not apply this path to CephFS

## Observability

Add explicit events and metrics around intent decisions:

- intent created/updated
- desired node changed
- same-node reuse selected
- maintenance hold selected
- cross-node move rejected
- OpenNebula attach observed
- node device discovered
- recovery action selected
- recovery action succeeded/failed
- stale intent cleared

Useful metrics:

- `opennebula_csi_attachment_intent_total{operation,outcome,reason}`
- `opennebula_csi_attachment_intent_state{phase}`
- `opennebula_csi_attachment_intent_action_total{action,outcome}`
- `opennebula_csi_attachment_intent_residency_seconds{phase}`
- `opennebula_csi_attachment_intent_generation{volume,node}`

Support bundle should include:

- intent ledger
- current desired vs observed state
- last selected action and reason
- OpenNebula attachment snapshot
- node device reports
- maintenance state
- metadata drift quarantine state
- host artifact quarantine state
- exact operator repair targets: image ID, VM ID, disk ID, target device, and node

## Implementation phases

### Phase 0: Close current incident

Do not start the architectural refactor until the current cluster is fully reconciled.

Exit criteria:

- no stuck deleting `VolumeAttachment` objects
- no local RWO PVCs stuck in attach/detach retry loops
- OpenNebula shows no unexpected HOTPLUG activity
- affected workloads are running or intentionally rescheduled
- `v0.5.12` metadata drift and quarantine behavior is validated from controller logs and support-bundle output
- `v0.5.13` host-artifact conflict detection validates stale local LV failures as read-only `FailedPrecondition` states with support-bundle output
- `opennebula-cli` has a tracked handoff for VM disk, recover, reboot, wait, and richer show coverage needed during repair workflows

### Phase 1: Read-only intent shadowing

Build a read-only intent collector that computes intent from existing Kubernetes/OpenNebula state but does not drive mutations.

Deliverables:

- in-memory model
- support-bundle output
- events or logs for selected hypothetical action
- metrics for drift classes

Goal:

- verify the decision algorithm against live clusters without changing behavior.

### Phase 2: ConfigMap-backed intent ledger

Persist intent state in a ConfigMap.

Deliverables:

- generation tracking
- stale generation discard
- same-node hold decisions
- integration with current hotplug queue
- support-bundle inclusion

Goal:

- use the ledger for safer decisions while avoiding CRD migration risk.

### Phase 3: Reconciler-controlled local RWO actions

Move local RWO attach/detach decisions into the reconciler.

Deliverables:

- publish/unpublish update intent first
- reconciler owns physical detach/attach ordering
- same-node reuse and maintenance holds are native actions
- same-node device recovery is driven from intent state

Goal:

- reduce unnecessary OpenNebula mutations during rollouts.

### Phase 4: CRD promotion

Promote the ledger to `OpenNebulaAttachmentIntent`.

Deliverables:

- CRD schema
- status conditions
- kubectl-friendly columns
- inventory controller summary
- migration from ConfigMap state

Goal:

- make attachment state inspectable and supportable by operators.

### Phase 5: Policy and automation

Add policy knobs after the core model is stable.

Candidate policies:

- maximum maintenance hold duration
- cross-node local RWO movement policy: `never`, `afterTimeout`, `operatorApproved`
- node upgrade detection policy
- stale intent garbage collection
- workload class overrides

## Acceptance criteria for v0.6.0

Minimum acceptable release candidate:

- read-only intent shadowing available in support bundles
- local RWO desired vs observed state visible per volume
- decision algorithm classifies current attach/detach loops correctly
- no behavioral change unless explicitly enabled

Preferred release candidate:

- ConfigMap-backed ledger enabled behind a feature flag
- local RWO same-node hold decisions driven by the ledger
- `v0.5.11` device recovery integrated as a reconciler action
- rollback path is disabling the feature flag and clearing the ConfigMap

## Risks

- incorrect desired-state inference could block legitimate rescheduling
- persistent intent state can become stale if not generation-scoped
- CRD promotion too early may lock in the wrong schema
- adding another controller loop can obscure CSI RPC behavior if logging is weak
- cross-node local RWO policy must be conservative to avoid data-loss risk

## Open questions

- Should `AttachmentIntent` be driver-internal only at first, or operator-visible from the start?
- What is the default maximum hold time for local RWO during node maintenance?
- Should cross-node local RWO movement require an annotation or explicit operator action?
- How should the driver detect Talos/Omni upgrade intent without hard-coding Omni-specific behavior?
- Should node-side stage success be written directly to intent status, or continue through ConfigMap reports until the CRD exists?

## Current recommendation

Finish validating the `v0.5.12` metadata drift guard first. Then start `v0.6.0` with read-only intent shadowing and support-bundle output. That gives us production data for the algorithm before allowing it to control OpenNebula attach/detach mutations.
